§1 第一部分：各家大模型注意力机制的发展脉络

注意力机制的演进主线，是一部"与显存和显存带宽（Memory Bandwidth）作斗争"的历史。在解码阶段，模型每次计算都需要读取历史上所有的 KV Cache，这导致推理速度被显存读取速度死死卡住（Memory Bound）。

统一符号表（以 LLaMA-3 8B 为例）

符号含义LLaMA-3 8B
$d$模型隐藏维度4096
$H$Query 头数32
$H_{kv}$KV 头数（GQA 时 $< H$）8
$d_h$每头维度 $= d / H$128
$G$每组 Q 头数 $= H / H_{kv}$4
$N$序列长度—
$L$层数32

1.1 经典原点：MHA (Multi-Head Attention)

  • 代表模型： Transformer 原作, GPT-3, LLaMA-1, OPT, Bloom
  • 机制原理： 每个 Query 头都有自己独立的、对应的 Key 和 Value 头。$H_{kv} = H$。

1.1.1 MHA 公式推导

第一步：线性投影——将输入 $X \in \mathbb{R}^{N \times d}$ 投影到 $H$ 个独立的 QKV 子空间：

$$Q_h = X \, W^Q_h, \quad K_h = X \, W^K_h, \quad V_h = X \, W^V_h \qquad h = 1, \ldots, H$$ $W^Q_h, W^K_h, W^V_h \in \mathbb{R}^{d \times d_h}$，每个头有独立的 K、V 投影

第二步：Scaled Dot-Product Attention（$M$ 为因果掩码）：

$$A_h = \text{softmax}\!\left(\frac{Q_h \, K_h^\top}{\sqrt{d_h}} + M \right) \quad \in \mathbb{R}^{N \times N}$$

第三步：加权聚合 + 拼接输出：

$$\text{MHA}(X) = [\text{head}_1 \mid \cdots \mid \text{head}_H] \, W^O, \qquad \text{head}_h = A_h \, V_h$$

KV Cache（每层每 token）：$\text{KV}_{\text{MHA}} = 2 \times H \times d_h = 2d$

1.1.2 MHA PyTorch 核心代码

import torch, torch.nn as nn, torch.nn.functional as F, math

class MultiHeadAttention(nn.Module):
    """标准 Multi-Head Attention (MHA)"""
    def __init__(self, d_model: int, n_heads: int):
        super().__init__()
        self.n_heads = n_heads
        self.d_head = d_model // n_heads

        # 每个头有独立的 Q, K, V 投影
        self.W_q = nn.Linear(d_model, d_model, bias=False)  # [d, H * d_h]
        self.W_k = nn.Linear(d_model, d_model, bias=False)  # [d, H * d_h]
        self.W_v = nn.Linear(d_model, d_model, bias=False)  # [d, H * d_h]
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, N, D = x.shape

        # ① 线性投影 → [B, N, H, d_h] → [B, H, N, d_h]
        q = self.W_q(x).view(B, N, self.n_heads, self.d_head).transpose(1, 2)
        k = self.W_k(x).view(B, N, self.n_heads, self.d_head).transpose(1, 2)
        v = self.W_v(x).view(B, N, self.n_heads, self.d_head).transpose(1, 2)

        # ② Scaled Dot-Product Attention + 因果掩码
        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(self.d_head)
        causal_mask = torch.triu(
            torch.full((N, N), float('-inf'), device=x.device), diagonal=1
        )
        scores = scores + causal_mask
        attn = F.softmax(scores, dim=-1)  # [B, H, N, N]

        # ③ 加权聚合 + 拼接
        out = torch.matmul(attn, v)  # [B, H, N, d_h]
        out = out.transpose(1, 2).contiguous().view(B, N, D)
        return self.W_o(out)

1.2 极端压缩：MQA (Multi-Query Attention)

  • 代表模型： PaLM, Falcon, StarCoder
  • 机制原理： $H$ 个 Query 头共享唯一 1 个 Key 头和 Value 头。$H_{kv} = 1$。

1.2.1 MQA 公式推导

与 MHA 唯一的区别：K 和 V 的投影矩阵只有一份，不再按头区分：

$$Q_h = X \, W^Q_h \;\;(h=1..H), \qquad K = X \, W^K, \qquad V = X \, W^V$$ $W^K, W^V \in \mathbb{R}^{d \times d_h}$——注意没有下标 $h$

注意力（所有头共用同一个 K 和 V）：

$$A_h = \text{softmax}\!\left(\frac{Q_h \, K^\top}{\sqrt{d_h}} + M\right), \qquad \text{head}_h = A_h \, V$$

KV Cache（每层每 token）：$\text{KV}_{\text{MQA}} = 2 \times 1 \times d_h = 2d_h = \frac{2d}{H}$，对比 MHA 压缩了 $H$ 倍。

1.2.2 MQA PyTorch 核心代码

class MultiQueryAttention(nn.Module):
    """Multi-Query Attention (MQA) — 所有 Q 头共享 1 个 KV 头"""
    def __init__(self, d_model: int, n_heads: int):
        super().__init__()
        self.n_heads = n_heads
        self.d_head = d_model // n_heads

        self.W_q = nn.Linear(d_model, d_model, bias=False)     # H 个 Q 头
        self.W_k = nn.Linear(d_model, self.d_head, bias=False)  # ← 只有 1 个 K 头！
        self.W_v = nn.Linear(d_model, self.d_head, bias=False)  # ← 只有 1 个 V 头！
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, N, D = x.shape
        q = self.W_q(x).view(B, N, self.n_heads, self.d_head).transpose(1, 2)
        # k, v: [B, N, 1, d_h] → 广播到 [B, H, N, d_h]
        k = self.W_k(x).unsqueeze(1)  # [B, 1, N, d_h]
        v = self.W_v(x).unsqueeze(1)  # [B, 1, N, d_h]

        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(self.d_head)
        causal_mask = torch.triu(
            torch.full((N, N), float('-inf'), device=x.device), diagonal=1
        )
        attn = F.softmax(scores + causal_mask, dim=-1)
        out = torch.matmul(attn, v)  # 广播: [B, H, N, N] × [B, 1, N, d_h]
        out = out.transpose(1, 2).contiguous().view(B, N, D)
        return self.W_o(out)
💡 关键区别只有一处：W_k 和 W_v 的输出维度从 d_model（$= H \times d_h$）变成了 d_head（$= 1 \times d_h$）。KV Cache 体积直接缩小 $H$ 倍。

1.3 甜点均衡：GQA (Grouped-Query Attention)

目前行业绝对主流
  • 代表模型： LLaMA-2/3, Mistral, Qwen-1.5/2, Gemma
  • 机制原理： 将 $H$ 个 Query 头分成 $H_{kv}$ 组，每组 $G = H/H_{kv}$ 个 Q 头共享一个 KV 头。

1.3.1 GQA 公式推导

投影：

$$Q_h = X \, W^Q_h \;\;(h=1..H), \qquad K_g = X \, W^K_g, \;\; V_g = X \, W^V_g \;\;(g=1..H_{kv})$$

分组共享规则（LLaMA-3 8B：$H=32, H_{kv}=8, G=4$）：

$$\text{第 } h \text{ 个 Q 头，使用第 } g = \lfloor h / G \rfloor \text{ 组的 KV}$$ Q 头 0,1,2,3 → KV 组 0；Q 头 4,5,6,7 → KV 组 1；…

注意力：

$$A_h = \text{softmax}\!\left(\frac{Q_h \, K_{g(h)}^\top}{\sqrt{d_h}} + M\right), \qquad \text{head}_h = A_h \, V_{g(h)}$$

KV Cache（每层每 token）：$\text{KV}_{\text{GQA}} = 2 \times H_{kv} \times d_h$。LLaMA-3 8B：$2 \times 8 \times 128 = 2048$，对比 MHA 的 $8192$，压缩 4 倍。

1.3.2 GQA PyTorch 核心代码

class GroupedQueryAttention(nn.Module):
    """Grouped-Query Attention (GQA) — 当前行业主流"""
    def __init__(self, d_model: int, n_heads: int, n_kv_heads: int):
        super().__init__()
        self.n_heads = n_heads        # Q 头数 (32)
        self.n_kv_heads = n_kv_heads  # KV 头数 (8)
        self.n_groups = n_heads // n_kv_heads  # 每组 Q 头数 (4)
        self.d_head = d_model // n_heads

        self.W_q = nn.Linear(d_model, n_heads * self.d_head, bias=False)
        self.W_k = nn.Linear(d_model, n_kv_heads * self.d_head, bias=False)  # 只有 8 个 KV 头
        self.W_v = nn.Linear(d_model, n_kv_heads * self.d_head, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, N, D = x.shape
        q = self.W_q(x).view(B, N, self.n_heads, self.d_head).transpose(1, 2)
        k = self.W_k(x).view(B, N, self.n_kv_heads, self.d_head).transpose(1, 2)
        v = self.W_v(x).view(B, N, self.n_kv_heads, self.d_head).transpose(1, 2)

        # ★ 核心：将 KV 头 repeat 到与 Q 头数量对齐 (8 → 32)
        k = k.repeat_interleave(self.n_groups, dim=1)  # [B, 32, N, 128]
        v = v.repeat_interleave(self.n_groups, dim=1)  # [B, 32, N, 128]

        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(self.d_head)
        causal_mask = torch.triu(
            torch.full((N, N), float('-inf'), device=x.device), diagonal=1
        )
        attn = F.softmax(scores + causal_mask, dim=-1)
        out = torch.matmul(attn, v)
        out = out.transpose(1, 2).contiguous().view(B, N, D)
        return self.W_o(out)
💡 核心秘密：repeat_interleave 在计算时把 8 个 KV 头扩展成 32 个（零成本广播），但在存储 KV Cache 时只需保存 8 份。计算时每个 Q 头都"看到了完整信息"，存储时只付 1/4 的代价。

1.4 潜空间压缩：MLA (Multi-head Latent Attention)

  • 代表模型： DeepSeek-V2, DeepSeek-V3, DeepSeek-R1
  • 核心思想： 与其直接存 K 和 V（维度很高），不如先"压缩"再"解压"。

1.4.1 MLA 公式推导

第一步：下投影（Compress）——将输入压缩成低维隐向量

$$c_t = X_t \, W^{DKV} \quad \in \mathbb{R}^{d_c}$$ $W^{DKV} \in \mathbb{R}^{d \times d_c}$，$d_c \ll d$（DeepSeek-V2: $d=5120, d_c=512$，压缩比 10:1）
★ 这个 $c_t$ 就是唯一需要缓存的东西！KV Cache 从存 K+V（$2 \times d$）变为只存 $c_t$（$d_c$）。

第二步：上投影（Decompress）——从隐向量实时还原出 K 和 V

$$K_h = c_t \, W^{UK}_h, \qquad V_h = c_t \, W^{UV}_h$$ $W^{UK}_h, W^{UV}_h \in \mathbb{R}^{d_c \times d_h}$——每次注意力计算时实时完成，不缓存

第三步：RoPE 位置编码的解耦

传统注意力中，RoPE 直接加在 K 上。但 MLA 的 K 是从 $c_t$ 还原的——如果把 RoPE 直接加到 $c_t$ 上，位置信息会"污染"隐向量。

DeepSeek 的天才解法——解耦 RoPE：

$$K_h^{\text{final}} = \big[\underbrace{c_t \, W^{UK}_h}_{K^{\text{nope}}_h \text{（内容）}} \;\big|\; \underbrace{\text{RoPE}(X_t \, W^{KR})}_{K^{\text{rope}} \text{（位置）}}\big]$$ 内容信息从隐向量还原（不加 RoPE），位置信息从原始输入直接投影（加 RoPE），两者 concat

KV Cache（每层每 token）：$\text{KV}_{\text{MLA}} = d_c + d_r$。DeepSeek-V2: $512 + 64 = 576$，对比 MHA 的 $2 \times 5120 = 10240$，压缩约 18 倍。

1.4.2 MLA PyTorch 核心代码

class MultiHeadLatentAttention(nn.Module):
    """Multi-head Latent Attention (MLA) — DeepSeek-V2/V3 核心
       KV Cache 只需存 [compressed_kv, rope_key]，极致压缩"""
    def __init__(self, d_model: int, n_heads: int,
                 d_compress: int, d_rope: int):
        super().__init__()
        self.n_heads = n_heads
        self.d_head = d_model // n_heads
        self.d_compress = d_compress
        self.d_rope = d_rope

        # Q 投影（标准）
        self.W_q = nn.Linear(d_model, d_model, bias=False)
        # ★ 下投影：x → 低维隐向量 c（KV Cache 的主体）
        self.W_dkv = nn.Linear(d_model, d_compress, bias=False)
        # ★ 上投影：c → 高维 K, V（推理时实时计算，不缓存）
        self.W_uk = nn.Linear(d_compress, d_model, bias=False)
        self.W_uv = nn.Linear(d_compress, d_model, bias=False)
        # ★ 解耦 RoPE：单独投影出小维度 key（需要额外缓存）
        self.W_kr = nn.Linear(d_model, d_rope, bias=False)
        # 输出
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def apply_rope(self, x):
        """简化版 RoPE（实际实现用 cos/sin 旋转）"""
        return x  # placeholder

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, N, D = x.shape

        # ── Q 路径（标准）──
        q = self.W_q(x).view(B, N, self.n_heads, self.d_head).transpose(1, 2)

        # ── KV 路径（MLA 核心）──
        c = self.W_dkv(x)  # [B,N,d_c] ← ★ KV Cache 存这个

        # 上投影 → 实时还原 K, V（不缓存）
        k_nope = self.W_uk(c).view(B, N, self.n_heads, self.d_head).transpose(1, 2)
        v = self.W_uv(c).view(B, N, self.n_heads, self.d_head).transpose(1, 2)

        # 解耦 RoPE → 额外小维度 key
        k_rope = self.apply_rope(self.W_kr(x))  # [B,N,d_r]

        # ── Attention ──
        scores = torch.matmul(q, k_nope.transpose(-2, -1)) / math.sqrt(self.d_head)
        causal_mask = torch.triu(
            torch.full((N, N), float('-inf'), device=x.device), diagonal=1
        )
        attn = F.softmax(scores + causal_mask, dim=-1)
        out = torch.matmul(attn, v)
        out = out.transpose(1, 2).contiguous().view(B, N, D)
        return self.W_o(out)

    def get_kv_cache(self, x):
        """推理时：只返回需要缓存的内容"""
        c = self.W_dkv(x)                       # [B,N,d_c] ← 主体
        k_rope = self.apply_rope(self.W_kr(x))   # [B,N,d_r] ← 位置
        return c, k_rope  # ← 这就是全部 KV Cache！

1.5 架构革命：线性注意力与无 KV Cache 架构

  • 代表模型： Mamba (SSM), RWKV (RNN), Jamba, GLA
  • 机制原理： 传统 Attention 的复杂度是 $O(N^2)$，且 KV Cache 随长度增长。这类架构通过状态空间方程或门控线性注意力，将所有历史信息压缩为一个固定大小的隐状态（Hidden State）。
  • 现状： 推理阶段彻底消灭了 KV Cache，显存占用恒定。但目前在"大海捞针"和 In-Context Learning 等任务上，还未能完全超越标准 Transformer。

各方案 KV Cache 显存终极对比

以 $d = 4096, H = 32, d_h = 128, H_{kv} = 8$ 为例，FP16 精度（2 bytes/param），单层单 token：

方案KV Cache 维度每层每token字节相对MHA代表模型

💡 实际场景计算：32K 上下文、32 层、Batch=64 的 LLaMA-3 8B (GQA)：
KV Cache 总占用 = $64 \times 32000 \times 32 \times 2048 \times 2 \approx \mathbf{8.4 \text{ GB}}$
如果用 MHA 则需 $\approx \mathbf{33.6 \text{ GB}}$——几乎占满一张 A100 40GB！

§2 第二部分：现代端到端推理流程的系统级变革

你列出的标准流程在逻辑上依然是成立的，这是所有文本生成任务的算法基石：

  1. Prompt → 2. Tokenizer → 3. Prefill (算全段KV) → 4. Decode (自回归) → 5. Stop → 6. Detokenize

然而，在现代真实的工业生产环境（如 vLLM, TensorRT-LLM, SGLang 等推理引擎）中，物理执行层面已经发生了翻天覆地的变化。为了极致压榨 GPU 性能，流程中的每一步都被高度魔改了。

变革 1：内存管理的革命 —— PagedAttention

标准流程痛点： 预先为每个句子的 KV Cache 分配连续的显存空间。这会导致大量的显存碎片和预留浪费，利用率极低（常低于30%）。
现代引擎突围： 引入 PagedAttention。像操作系统管理虚拟内存一样，将 KV Cache 切分为固定大小的 Block（如每块 16 个 token）。在 Decode 阶段按需分配物理显存，块与块之间物理上不连续。这彻底解决了 OOM 问题，并将系统吞吐量翻了数倍。

变革 2：调度机制的革命 —— Continuous Batching (In-flight Batching)

标准流程痛点： 静态批处理（Static Batching）。必须等 Batch 里所有句子都走到 Stop 阶段才能接下一批。短句子要干等长句子，GPU 算力严重闲置。
现代引擎突围： 连续批处理（Continuous Batching）。以单步 Iteration 为调度单位。若池子里有请求触发了 EOS，立刻释放资源，调度器瞬间将等待中的新请求（Prefill）塞入当前 Batch。在 GPU 内部，Prefill 序列和 Decode 序列在同一个 Batch 中混合计算。

变革 3：Prefill 阶段的革命 —— Chunked Prefill & Prefix Caching

面对超长上下文，一次性 Prefill 会导致计算尖峰和 OOM。

  • Chunked Prefill（分块预填充）： 引擎将 100K 长的 Prompt 强行切碎（比如 4K 一块），分多次 Prefill 塞进不同的 Batch 算完。这平滑了系统的算力开销，避免卡死其他用户的 Decode。
  • Prefix Caching（前缀缓存/基数树）： 如 SGLang 维护一棵 Radix Tree，若用户请求带有相同的 System Prompt 或历史对话，引擎会从内存直接"命中"并复用 KV Cache，彻底跳过该段的 Prefill 计算，实现首字延迟（TTFT）接近 0。

变革 4：Decode 阶段的革命 —— Speculative Decoding (投机解码)

Decode 是严格的自回归操作，每次算 1 个 token 导致 GPU 的海量计算单元（Tensor Cores）无事可做，沦为内存搬运工。

Speculative Decoding 打破自回归步长限制
  1. 挂载一个小规模"草稿模型"，飞速往前"猜"出 $N$ 个候选 token（如 [A, B, C, D]）。
  2. 大模型（目标模型）把这 $N$ 个 token 拼接起来，当作一句话一次性并行验证（类似 Prefill 机制）。
  3. 若验证通过，大模型相当于做了一次矩阵运算就往前走了 4 步，解码速度成倍跃升。

变革 5：输出结构的革命 —— Structured Decoding (结构化解码)

在 Decode 产生 Logits 到采样输出这个环节之间，插入了状态机约束（如 XGrammar / Outlines）。当大模型生成的 Logits 中包含会导致 JSON 等语法错误的 token 时，引擎会将其概率强行置为 $-\infty$（负无穷）。这确保了模型输出 100% 合法，将 Decode 从"自由创作"变成了"戴着镣铐跳舞"。

§3 全局因果链总结
