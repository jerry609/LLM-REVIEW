# 面试题：RAG 系统的 Serving 优化

## 题目
"你的 RAG 系统每次请求需要检索 10 段文档拼接到上下文中。上线后 TTFT 很高，怎么优化？"

## 答题思路

### 问题分析
- RAG 请求特点：输入长（检索结果 + query），输出相对短
- TTFT 高的原因：① prefill 计算量大（长输入）；② 无法利用前缀缓存（检索结果每次不同）

### 优化方案

#### 1. 前缀缓存优化
- System prompt 部分是固定的 → 可以缓存
- 检索结果按相似度排序 → 提高部分前缀复用概率
- 热门文档 chunk 的 KV 预计算并缓存

#### 2. 减少输入 token 数
- 检索结果压缩：只取关键句子而非整段
- Reranker 过滤：从 10 段减到 3-5 段
- Prompt 模板压缩：去除冗余指令

#### 3. Prefill 加速
- Chunked Prefill → 不阻塞其他请求
- P/D 分离 → Prefill 专用 GPU
- FlashAttention → 减少 IO

#### 4. 架构级优化
- Streaming 返回：边生成边返回，用户感知 TTFT 降低
- 异步检索 + 投机：在检索结果返回前先用 query 开始 prefill
- 分级回答：先给快速简答，再给详细回答

## 面试一句话
- "RAG 的 TTFT 瓶颈在长输入 prefill。优化方向：缓存固定前缀、压缩检索结果、P/D 分离、异步流水线。"
