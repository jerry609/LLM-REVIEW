# Q004: Prefill/Decode 与 PagedAttention 性能优化题

## 题干
你接手的推理服务在压测中表现如下：
- TTFT 偏高（首token慢）
- TPOT 波动大（生成阶段不稳定）
- 显存还有余量，但吞吐上不去

当前系统使用连续批处理（continuous batching），但 KV 块管理较粗糙。请你设计优化方案，重点解释：
1. Prefill 与 Decode 的资源瓶颈有何不同？
2. 为什么 PagedAttention/块化管理能改善吞吐与碎片问题？
3. 如何设计块大小、分配/回收策略？
4. 调度器如何在 TTFT 与 TPOT 间权衡？
5. 你会如何验证优化有效并避免回归？

## 期望考点
- Prefill（算力受限）与 Decode（访存受限）阶段差异
- PagedAttention 的页表映射、块复用、低碎片优势
- 连续批处理 + 动态调度策略
- 端到端指标闭环（TTFT/TPOT/吞吐/P99）

## 参考答案（示例）
- **阶段差异**
  - Prefill：并行度高、计算密集，主要受 GPU compute 限制；
  - Decode：每步token少但步数多，主要受 memory bandwidth 与调度开销影响。
- **PagedAttention 价值**
  - 将 KV 拆分为固定块，逻辑连续、物理可离散，减少大块连续内存需求；
  - 降低显存碎片，支持按块分配/回收/迁移，提升并发承载能力。
- **块参数建议**
  - 块太小：页表与调度开销增大；
  - 块太大：内部碎片增加、回收不灵活；
  - 需通过压测选取折中点（常见做法：按模型与请求长度分布做网格搜索）。
- **调度策略**
  - TTFT优先：给新请求Prefill更高短期优先级；
  - 吞吐优先：增大Decode批次，减少kernel launch开销；
  - 折中：设置双队列与配额（Prefill最小保障 + Decode最大批次上限）。
- **验证方法**
  - 同一流量回放下对比优化前后：TTFT/TPOT/P95/P99/吞吐/GPU利用率；
  - 观察块碎片率、块分配失败率、回收延迟；
  - 设置回归阈值：若P99超阈或TTFT恶化则自动回滚。
