# Q002: KV驱逐策略深挖（LRU / Priority / H2O）

## 题干
你维护一个多租户LLM推理平台，GPU KV缓存总容量固定。当前现象：
- 高峰期频繁触发驱逐，P99延迟抖动明显；
- 大上下文用户会挤占空间，小请求延迟被拖慢；
- 业务希望“吞吐不降、质量不掉、且公平性更好”。

请设计并对比以下三类策略，并给出上线方案：
1. 纯 LRU（块级）
2. 优先级驱逐（基于会话等级/预计重用/块热度）
3. H2O 类重token保留策略（heavy-hitter aware）

需要回答：
- 每种策略的适用场景、优缺点
- 如何定义驱逐分数（给出可落地公式）
- 如何避免“误驱逐后大量重算”
- 多租户下如何做公平性控制
- 如何灰度、监控与回滚

## 期望考点
- 驱逐策略设计：命中率、重算成本、延迟稳定性三者权衡
- 块级管理与请求级SLO约束
- 多租户配额/权重/限流协同
- 指标闭环：命中率、重算率、P95/P99、OOM率

## 参考答案（示例）
- **LRU（基线）**
  - 优点：实现简单，开销低；
  - 缺点：只看“最近性”，不看“未来重用价值”，高峰期容易误驱逐热点块。
- **优先级驱逐（推荐线上主策略）**
  - 对每个候选块计算：
    `Score = w1*Recency + w2*ReuseProb + w3*RecomputeCost + w4*TenantPriority - w5*AgePenalty`
  - 驱逐最低分；其中 `ReuseProb` 可由最近窗口命中率+前缀匹配深度估计，`RecomputeCost` 可近似为块token数×层数。
- **H2O类策略（增强）**
  - 核心思想：优先保留对后续注意力贡献高的“重token”相关KV，减少质量与重算损失；
  - 风险：实现复杂，需要额外统计或近似，在线计算成本需受控。
- **防误驱逐机制**
  - 设置“最小工作集保留阈值”（每活跃会话至少保留N块）；
  - 引入“冷却窗口”：刚回填/刚命中的块短时间不驱逐；
  - 驱逐预算限速：每秒最大驱逐块数，避免抖动。
- **多租户公平性**
  - 采用 `hard quota + burst credit`：每租户基础配额+可借用额度；
  - 超额租户先驱逐本租户冷块，再全局驱逐；
  - 对低优先级租户在拥塞时降级到更激进压缩而非直接抢占。
- **灰度上线**
  - 阶段1：影子打分（只打分不驱逐），验证分数稳定性；
  - 阶段2：10%流量A/B，对比 LRU；
  - 阶段3：全量并保留一键回退到LRU。

## 指标看板（必须有）
- 缓存层：KV命中率、驱逐率、误驱逐回填率、平均驻留时长
- 性能层：TTFT、TPOT、吞吐、P95/P99延迟
- 资源层：GPU显存占用、OOM率、回填队列长度
- 公平性：各租户P99、各租户配额使用率、配额争抢次数
