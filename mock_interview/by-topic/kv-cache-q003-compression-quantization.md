# Q003: KV压缩与量化深挖（FP8/INT8/INT4 + 分层策略）

## 题干
你负责把现有推理服务的并发提升 2 倍，但不能新增GPU。团队提议对KV缓存做压缩：
- 方案A：全量FP8
- 方案B：热块BF16 + 温块INT8 + 冷块INT4（分层）
- 方案C：只做驱逐，不做压缩

请你给出决策与上线方案，回答：
1. 如何评估“压缩收益 vs 质量风险”？
2. 哪些KV块可以安全压缩，哪些不建议压缩？
3. 量化粒度（per-tensor/per-head/per-channel）如何选择？
4. 回迁反量化带来的延迟抖动怎么控制？
5. 你如何定义上线门槛与回滚条件？

## 期望考点
- KV压缩核心目标：扩容有效上下文与并发，不显著伤害质量
- 分层缓存策略（热/温/冷）与数据生命周期
- 量化方案权衡：精度、吞吐、实现复杂度
- 线上稳定性：反量化预算、队列控制、降级策略

## 参考答案（示例）
- **决策原则**
  - 先做“分层压缩”而不是“全量强压缩”；
  - 热路径优先保精度（BF16/FP16），冷路径优先容量（INT8/INT4/offload）。
- **推荐方案**
  - 线上首发：`热块BF16 + 温块FP8/INT8 + 冷块INT4(或CPU offload)`；
  - 配合优先级驱逐，形成“先压缩、后驱逐”的双层保护。
- **块选择规则（示例）**
  - 不压缩：最近K步高频命中块、系统提示词公共前缀关键块；
  - 可轻压缩：中等热度历史块（FP8/INT8）；
  - 重压缩：长尾冷块（INT4），并限制单步回迁数量。
- **量化粒度建议**
  - `per-tensor`：简单快，精度损失可能较高；
  - `per-head/per-channel`：精度更稳，元数据与计算开销增加；
  - 实践上先从 INT8 per-channel 或 FP8 起步，再评估 INT4。
- **延迟抖动控制**
  - 每decode step设置反量化预算（token级/块级）；
  - 异步预取下一批可能命中的冷块；
  - 设置“抢救阈值”：回迁队列过长时临时降级压缩级别或限流。
- **上线门槛（示例）**
  - 质量：关键任务集准确率下降 <= 0.5%（或业务自定义阈值）；
  - 性能：吞吐提升 >= 25%，P99恶化 <= 10%；
  - 稳定性：OOM率不升，回迁队列长度稳定。

1. 为什么有时 INT4 显存省很多，但实际吞吐不升反降？
2. 量化后命中率上升但质量下降，问题通常在哪？
3. 你如何做“问题定位”：量化误差 vs 调度抖动 vs 驱逐过猛？
4. 对代码生成、数学推理、对话任务，压缩敏感性是否一致？
5. 如果业务要求“零质量损失”，你会怎么谈判技术方案？
