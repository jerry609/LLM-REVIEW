# KV 驱逐策略面试问答

## Q1: 你用什么策略做 KV 驱逐？为什么？
**A**: 默认用 **LRU**（简单、O(1)、适合时间局部性）。如果流量有稳定热点，切 LFU；如果是多租户场景，叠加配额公平驱逐。线上建议先跑 LRU baseline 看命中率，再按需升级。

## Q2: LRU 和 LFU 各自的问题是什么？
**A**:
- LRU：热点漂移时反应慢（刚被访问但未来不再需要的条目占据位置）
- LFU：新条目 use_count=1 容易被立刻驱逐（"新手惩罚"），需要做衰减或窗口 LFU

## Q3: 多租户下大客户挤占小客户怎么办？
**A**: 给每个租户设配额（按权重分配总块数），驱逐时优先从超配租户中选 victim。用 Jain fairness index 监控：>0.9 表示公平。

## Q4: 怎么检测驱逐抖动？
**A**: 监控指标：① 短时间内同一 prefix 被驱逐又重新加载的次数；② 每请求平均驱逐数；③ KV 命中率突降。如果出现，说明工作集 > 缓存容量，需要扩容或更智能的驱逐策略。

## Q5: 驱逐粒度选块级还是 token 级？
**A**: 常规前缀缓存用**块级**（整个前缀的 block 一起驱逐），实现简单。H2O/SnapKV 等做 **token 级**稀疏（只保留重要 token），压缩率更高但实现复杂度也更高。

## Q6: 如果让你设计一个自适应驱逐策略，你怎么做？
**A**: 维护滑动窗口监控 hit rate 和 fairness，每隔 N 步评估一次：
- hit rate < 阈值 → 切 LFU（强化频率信号）
- fairness < 阈值 → 切 Fair（保护小租户）
- 两者都正常 → 回到 LRU（简单高效）
